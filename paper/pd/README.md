# PD Paper Workspace

This workspace contains Parkinson's disease paper analysis code built on top of
the outputs generated by `lfptensorpipe`.

## Input Layout

Feature tables are read from:

```text
{project}/derivatives/lfptensorpipe/sub-*/{record}/features/{trial}/...
```

For the current dataset, this includes paths such as:

```text
{project}/derivatives/lfptensorpipe/sub-010/sit/features/off/periodic/mean-scalar.pkl
{project}/derivatives/lfptensorpipe/sub-010/gait/features/walk/periodic/mean-scalar.pkl
```

## Anatomy Table Workflow

Anatomy summary tables are read from:

```text
{project}/derivatives/lfptensorpipe/sub-*/sit/localize/channel_representative_coords.pkl
```

The channel-coordinate export writes a cohort-level CSV to:

```text
{project}/summary/cohort/channel_coords.csv
```

The exported table keeps only these columns:

```text
Subject, Channel, Region, MNI_x, MNI_y, MNI_z
```

`Region` is recomputed from the localization flags:

- `SNr_in=True`, `STN_in=True` -> `Mid`
- `SNr_in=True`, `STN_in=False` -> `SNr`
- `SNr_in=False`, `STN_in=True` -> `STN`
- `SNr_in=False`, `STN_in=False` -> `EXT`

IPython example:

```python
from paper.pd.anat.anat import build_channel_coords_table, export_channel_coords_table

table = build_channel_coords_table()
output_path = export_channel_coords_table()
table.head()
```

You can also run the module directly from IPython:

```python
%run paper/pd/anat/anat.py
```

## Merge Goals

The merge workflow exports only named paper-level summary tables. Each named
merge combines selected `record -> trials` groups into a flattened output path
for downstream statistics and visualization.

Default project-specific configuration lives in `paper/pd/specs.py`, including
the default project root and interactive merge settings.

## Merge Specification

Named merges are configured as:

```python
Dict[str, Dict[str, str | Sequence[str]]]
```

Example:

```python
{
    "cycle": {"gait": "cycle_l"},
    "med": {"sit": "off", "sit_medon": "on"},
    "motor": {"gait": ["stand", "walk"]},
    "turn_stack": {"gait": "turn_stack"},
    "turn": {"gait": "turn"},
}
```

Default interactive values such as the project root, merge spec, and strict
selection mode are centralized in `paper/pd/specs.py`.

Implementation notes:

- Each top-level key is the exported merge name.
- Inner keys are source records.
- Inner values are one trial or a sequence of trials.
- Trial matching is tolerant to `_` versus `-` so a spec value such as
  `turn_stack` can match a source folder named `turn-stack`.

## Output Layout

The merge workflow writes to:

```text
{project}/summary/table/
```

Named merge outputs:

```text
{project}/summary/table/{name}/periodic/mean-scalar.pkl
```

Scalar outputs also write a sibling `.xlsx` file:

```text
{project}/summary/table/{name}/periodic/mean-scalar.xlsx
```

Each merged output is simplified before concatenation so downstream paper
analysis uses a compact, path-oriented schema. Merged tables keep only:

```text
Subject, Channel, Side, MNI_x, MNI_y, MNI_z, Region, Band, Phase, Lat, Value
```

Additional merge rules:

- rows with a top-level missing `Value` are dropped
- nested `Value` cells (`pd.Series` / `pd.DataFrame`) are kept even when they
  contain internal `NaN` values
- string values, columns, and index labels are normalized with first-letter
  capitalization for display consistency
- connectivity outputs keep the original `Channel` tuple and tuple-valued MNI
  coordinates
- `Side` is inferred from `Channel`
  - single channels: `0_1 / 1_2 / 2_3 -> L`, `8_9 / 9_10 / 10_11 -> R`
  - channel pairs use the first channel to determine `Side`

Region filtering is metric-specific:

- non-connectivity metrics (`aperiodic`, `periodic`, `raw_power`, `burst`)
  keep only `SNr` and `STN` rows:
  `SNr_in=True, STN_in=False -> SNr`
  `SNr_in=False, STN_in=True -> STN`
- unordered connectivity metrics (`coherence`, `plv`, `pli`, `ciplv`, `wpli`)
  keep only `SNr-STN` rows:
  `SNr-STN_in=True, SNr-SNr_in=False, STN-STN_in=False -> SNr-STN`
- ordered connectivity metrics (`trgc`, `psi`) keep only `SNr->STN` rows:
  `SNr-STN_in=True, STN-SNr_in=False, SNr-SNr_in=False, STN-STN_in=False -> SNr->STN`

Phase handling is merge-name specific:

- `cycle` splits `Phase` into `Phase` and `Lat`
  - `Phase` keeps only the event label such as `Strike` or `Off`
  - `Lat` compares the phase suffix (`L` / `R`) against the inferred `Side`
  - matching side gives `Lat=Ipsi`; the opposite side gives `Lat=Contra`
  - only `Strike` and `Off` are kept
- `turn` keeps only `Pre`, `Onset`, `Offset`, and `Post`
  - `Lat` is filled with missing values
- `turn_stack` keeps its existing phase values such as `All`
  - `Lat` is filled with missing values
- `med` rewrites `Phase` from the selected trial name at merge time
  - `trial=off -> Phase=Off`
  - `trial=on -> Phase=On`
  - `Lat` is filled with missing values
- `motor` rewrites `Phase` from the selected trial name at merge time
  - `trial=stand -> Phase=Stand`
  - `trial=walk -> Phase=Walk`
  - `Lat` is filled with missing values
- other merge names do not apply phase filtering
  - `Lat` is filled with missing values
- trace-like tables (`mean-trace`, `na-raw`) do not apply phase filtering
  - `cycle` still splits available phase labels into `Phase`
  - for `cycle` trace-like tables, `Lat` is derived directly from `Side`
    (`L -> Ipsi`, `R -> Contra`)
  - when trace-like tables do not carry a phase label, rows are kept and
    `Phase` stays missing

After simplification, merged tables are concatenated with `ignore_index=True`
and then `reset_index(drop=True)` to produce a clean row index.

## Preprocessing Workflow

Preprocessing for paper-level merged tables follows a fixed four-stage order:

1. summary
2. transform
3. normalize
4. shift

The preprocessing stage consumes named merge outputs under:

```text
{project}/summary/table/{name}/...
```

Summary outputs are written next to the source files with a `_summary` suffix:

```text
mean-trace.pkl -> mean-trace_summary.pkl
```

For scalar tables, summary also exports:

```text
mean-scalar.pkl -> mean-scalar_summary.xlsx
```

Summary uses `summarize_df` with:

```text
group_cols = Subject, Channel, Band, Phase, Lat
value_col = Value
mode = mean
drop_other_cols = False
```

Transform outputs are written next to the summary files with a `_trans` suffix:

```text
mean-trace_summary.pkl -> mean-trace_summary_trans.pkl
```

For scalar tables, transform also exports:

```text
mean-scalar_summary_trans.pkl -> mean-scalar_summary_trans.xlsx
```

Normalization always consumes transformed summary tables and writes a second
derived file with a `_trans_normalized` suffix:

```text
mean-trace_summary_trans.pkl -> mean-trace_summary_trans_normalized.pkl
```

For scalar tables, normalize also exports:

```text
mean-scalar_summary_trans_normalized.pkl -> mean-scalar_summary_trans_normalized.xlsx
```

This workspace should not write raw-only `_normalized` variants. The main suffix
chain is:

```text
raw -> _summary -> _summary_trans -> _summary_trans_normalized
```

`cycle` trace-like normalized tables also produce a shifted derivative:

```text
*_summary_trans_normalized.pkl -> *_summary_trans_normalized_shift.pkl
```

This shift stage applies only to `cycle` trace-like tables (`mean-trace`,
`na-raw`). For rows where `Lat == Ipsi`, the first half of the normalized time
axis is rolled to the second half, and `Lat` is rewritten to `Contra`. Rows
with `Lat == Contra` are kept unchanged.

Transform mode dispatch is path-based and uses the metric directory name
directly. For example, `burst/rate-scalar.pkl` uses the `burst -> rate` rule,
while `coherence/mean-trace.pkl` uses the `coherence` default rule without any
alias remapping.

Normalization is name-specific:

- `cycle`: baseline mean over `[0, 100]` percent
- `turn_stack`: baseline mean over `[0, 100]` percent
- `turn`: baseline mean over `[0, 25]` percent

Only trace-like nested tables (`mean-trace`, `na-raw`) are baseline-normalized.
Scalar and spectral tables use group-wise `normalize_df` only for selected
paper names:

- `cycle`: baseline `{"Phase": "Off"}`
- `turn`: baseline `{"Phase": "Pre"}`
- `med`: baseline `{"Phase": "Off"}`
- `motor`: baseline `{"Phase": "Stand"}`

For scalar and spectral normalization, the grouping columns are:

```text
Subject, Channel, Band, Lat
```

Other scalar and spectral tables still receive a `_summary_trans_normalized`
output as a pass-through copy of the transformed table so downstream scripts
can consume a consistent suffix convention.

When scalar or spectral normalization cannot find a matching baseline within a
group, that group is dropped instead of failing the whole file.

Every scalar `.pkl` written by merge or preprocessing also writes a matching
`.xlsx` file in the same directory.

## Statistics Inputs

R-based scalar statistics now run through a single configuration-driven entry
script:

```text
paper/pd/stats/run_scalar.R
```

This script consumes exported scalar workbooks directly from
`{project}/summary/table`, using an internal `ANALYSIS_SPECS` table to cover
the former cycle/med/motor/turn and single/double variants.

The shared scalar workbook inputs are:

```text
single metrics: raw_power / periodic / aperiodic / burst
double metrics: coherence / ciplv / pli / plv / psi / trgc / wpli
```

For each selected analysis spec, `run_scalar.R` should:

- collect `*scalar_summary_trans.xlsx` workbooks from the configured metric directories
- read workbook columns directly rather than legacy `region_raw.rds` files
- coerce `Value` back to numeric after Excel import
- expose `--jobs N`, defaulting to `parallel::detectCores(logical = TRUE)`
- treat each `(analysis spec, workbook)` pair as one independent parallel task
- keep `--jobs 1` as an explicit serial fallback for debugging or low-memory runs
- create one result folder per workbook under the metric directory
- export each per-band modeling subset as `input_band.xlsx`
- fit both `lmer` and `rlmer` models
- export omnibus tests, model diagnostics, and `EMM_*` CSV outputs

This single entry script preserves the existing output layout used by the
visualization scripts, but removes the duplicated per-section R scripts.

## Cycle Shared Preprocessing

Cycle-specific analysis now starts from one neutral preprocessing stage:

```text
paper/pd/preproc/cycle_trace_export.py
```

The exporter scans every shifted trace table under:

```text
{project}/summary/table/cycle/*/mean-trace_summary_trans_normalized_shift.pkl
```

It writes the shared preprocessed inputs to:

```text
{project}/summary/table/cycle/preprocessed/cycle_trace_long.csv
{project}/summary/table/cycle/preprocessed/cycle_trace_parameters.csv
```

The shared long-table columns are:

```text
Subject, Channel, Metric, Band, Region, t_pct, Value
```

This preprocessing stage exists to provide interval-ready cycle traces in this
repository.

Shared preprocessing should:

- include every metric that has a shifted `mean-trace` table
- keep every available band or aperiodic component as-is
- require all shifted rows to be in the `Contra` frame
- fill missing cycle points with circular linear interpolation before smoothing
- smooth each 500-point cycle trace with circular LOWESS using the same
  algorithmic settings as `smooth_trace` (`frac = 0.05`)
- downsample each smoothed trace with a 5-point block mean
- export a 100-bin cycle axis at bin centers `0.5, 1.5, ..., 99.5`

The supported metrics are discovered from the filesystem. For the current PD
workspace this includes:

```text
periodic
raw_power
aperiodic
coherence
ciplv
pli
plv
psi
trgc
wpli
```

## Cycle Interval Workflow

Cycle-specific zero-baseline interval analysis runs as a separate workflow:

```text
paper/pd/stats/run_cycle_interval_fit.R
paper/pd/stats/run_cycle_interval_postprocess.R
```

The interval fit stage reads only:

```text
{project}/summary/table/cycle/preprocessed/cycle_trace_long.csv
```

and writes:

```text
{project}/summary/table/cycle/interval/cycle_timepoint_deviation.csv
```

Because cycle trace-like tables are normalized to the full-cycle mean over
`[0, 100]%`, the pointwise null hypothesis is `Value = 0`.

Pointwise models are metric-class specific:

- local metrics (`aperiodic`, `periodic`, `raw_power`)
  - fit per `Metric/Band/t_pct`
  - `Value ~ Region + (1|Subject)`
  - keep `Region` inside the model and export one row per `Region` (`SNr`,
    `STN`) using region-specific mean-vs-zero contrasts
- connectivity metrics (`coherence`, `wpli`, `ciplv`, `pli`, `plv`, `psi`, `trgc`)
  - fit per `Metric/Band/Region/t_pct`
  - `Value ~ 1 + (1|Subject)`
  - export one row per connectivity `Region`

Pointwise outputs should include:

```text
Metric, Band, Region, t_pct,
lmer_estimate, lmer_p_value, lmer_status,
rlmer_estimate, rlmer_p_value, rlmer_status,
lmer_direction, rlmer_direction,
lmer_sig, rlmer_sig, both_sig_same_direction
```

The interval postprocess stage reads only:

```text
{project}/summary/table/cycle/interval/cycle_timepoint_deviation.csv
```

and writes:

```text
{project}/summary/table/cycle/interval/cycle_timepoint_lmer_intervals.csv
{project}/summary/table/cycle/interval/cycle_timepoint_lmer_interval_candidates.csv
{project}/summary/table/cycle/interval/cycle_timepoint_rlmer_intervals.csv
{project}/summary/table/cycle/interval/cycle_timepoint_rlmer_interval_candidates.csv
{project}/summary/table/cycle/interval/cycle_timepoint_joint_intervals.csv
{project}/summary/table/cycle/interval/cycle_timepoint_joint_interval_candidates.csv
```

The interval fit stage should group the shared preprocessed long table once and
then run pointwise models on those grouped slices, rather than repeatedly
rescanning the full CSV for every `Metric/Band/Region/t_pct` combination. It
should also support a `--jobs` CLI argument and default to
`parallel::detectCores(logical = TRUE)` so the pointwise fit stage can use the
full available CPU budget.

Interval postprocessing rules are:

- `lmer` basis
  - primary selector: `lmer_sig == TRUE`
  - direction source: `lmer_direction`
- `rlmer` basis
  - primary selector: `rlmer_sig == TRUE`
  - direction source: `rlmer_direction`
- `joint` basis
  - primary selector: `both_sig_same_direction == TRUE`
  - direction source: `lmer_direction`
- all three bases use signed intervals `above_0` or `below_0`
- minimum retained span: `5` contiguous bins (`5.0%` cycle)
- cycle is circular, so intervals that continue across `100 -> 0` must be
  merged into a single wrap-around interval

The 100-point cycle grid uses bin-center semantics, so interval boundaries
should be reported as bin edges:

- non-wrap interval
  - `start_pct = first_center - 0.5`
  - `end_pct = last_center + 0.5`
  - `span_pct = n_points`
- wrap-around interval
  - `start_pct = first_wrap_center - 0.5`
  - `end_pct = (last_wrap_center + 0.5) %% 100`
  - `span_pct = n_points`

For a wrap-around interval, `start_pct > end_pct` and `wraps_cycle = TRUE`.

Interval summaries should include:

```text
interval_basis, Metric, Band, Region, direction,
start_pct, end_pct, span_pct, wraps_cycle, n_points,
mean_estimate_lmer, min_p_lmer,
mean_estimate_rlmer, min_p_rlmer, robust_fraction
```

For all interval bases, `robust_fraction` is the proportion of bins inside the
retained interval that satisfy `both_sig_same_direction == TRUE`. This means the
`joint` interval outputs will always have `robust_fraction == 1.0` by
construction.

Candidate tables are filtered subsets of the full interval tables. They retain
only:

- `aperiodic` with `Band` in `Exponent`, `Offset`
- `periodic`
- `ciplv`
- `wpli`
- `coherence`
- `trgc`
- `psi`

Candidate ordering is basis-specific:

- `lmer`: `span_pct`, then `abs(mean_estimate_lmer)`, then `robust_fraction`
- `rlmer`: `span_pct`, then `abs(mean_estimate_rlmer)`, then `robust_fraction`
- `joint`: `span_pct`, then `abs(mean_estimate_lmer)`, then
  `abs(mean_estimate_rlmer)`, then `robust_fraction`

Old mixed-layout files under the previous cycle layout should be treated as
stale after this refactor. New runs in this repository should keep
`preprocessed/` and `interval/` outputs separate.

## Visualization Inputs

Visualization scripts now run through a single configuration-driven entry
script:

```text
paper/pd/viz/run_viz.py
```

This script consumes both paper tables and scalar statistics outputs from
`{project}/summary/table`, using an internal `VIZ_SPECS` table to cover the
former cycle/med/motor/turn/turn_stack and single/double variants.

The unified visualization entry handles four input families:

```text
scalar   -> {section}/{metric}/*scalar_summary_trans.xlsx
spectral -> {section}/{metric}/*spectral_summary_trans.pkl
trace    -> {section}/{metric}/*trace_summary_trans_normalized.pkl
raw      -> {section}/{metric}/*raw_summary_trans_normalized.pkl
```

Cycle shift visualizations add two more normalized families:

```text
trace_shift -> {section}/{metric}/*trace_summary_trans_normalized_shift.pkl
raw_shift   -> {section}/{metric}/*raw_summary_trans_normalized_shift.pkl
```

For scalar visualization specs, `run_viz.py` should consume the statistics
outputs written under the workbook-specific directory:

```text
{metric}/{workbook_stem}/{Band}/input_band.xlsx
{metric}/{workbook_stem}/{Band}/*_model/*_emm.csv
{metric}/{workbook_stem}/{Band}/*_model/*_tukey.csv
```

For spectral, trace, and raw visualization specs, `run_viz.py` should read the
paper workbook directly and write figures into the workbook-specific output
directory next to the input table.

All plotting `param_type` resolution should be table-driven rather than
branching on metric names, using either metric-only mappings (for
connectivity-style inputs) or `(metric, workbook_prefix)` mappings (for power
and burst families).

`run_viz.py` is the canonical paper-table visualization entrypoint. It should support:

```text
default      -> run every visualization spec
--section X  -> run all specs whose section matches X
--spec NAME  -> run one explicit spec
--jobs N     -> run workbook tasks with N worker processes
```

The supported sections are `cycle`, `med`, `motor`, `turn`, and `turn_stack`.
When no CLI filter is provided, the script should execute all configured
visualization specs in `VIZ_SPECS`.

Visualization execution should expand the selected specs into independent
workbook tasks, run those tasks with process-based parallelism by default, and
preserve `--jobs 1` as an explicit serial mode. Each worker should own one
workbook output directory at a time so figure paths and filenames remain
unchanged.

Cycle shift visualizations are exposed as additional specs:

```text
cycle_single_trace_shift
cycle_double_trace_shift
cycle_single_raw_shift
cycle_double_raw_shift
```

The shifted trace plots use `x_var="Region"`. The shifted raw plots use
`panel_var="Region"`.

Turn scalar and spectral figures should use the canonical phase order
`Pre -> Onset -> Offset -> Post` whenever `run_viz.py` plots `x_var="Phase"`
for the `turn` section. This ordering is a figure-layer constraint only: it
applies to the Python visualization wrappers and the final figure layout, but
does not rewrite `input_band.xlsx`, EMM CSVs, or Tukey CSVs generated by the R
statistics workflow. Turn raw and trace plots keep their existing grouping
behavior because they do not use `Phase` as the plotted grouping variable in the
canonical `run_viz.py` specs.

The plotting defaults for med visualization are migrated from MyLFP into
`paper/pd/viz/defaults.py`. This migrated file keeps the scalar plotting
configuration and wrapper functions, but excludes both the NIfTI-specific
`plot_triple_interaction_nifti` section and the `plot_*_fit` wrappers because
they are not part of the PD paper scalar workflow. The migrated defaults module
should import `lfptensorpipe.viz.visualdf` directly rather than relying on the
legacy `src.viz.visualdf` path. Visualization scripts should also reuse
`paper/pd/specs.py::DEFAULT_PROJECT_ROOT` instead of hard-coding the PD project
path. Tukey bracket rendering is delegated to `lfptensorpipe.viz.visualdf`, so
the stats CSV inputs are consumed without an extra paper-level p-value adapter.
The current visualdf compatibility fix for these plots uses a writable NumPy
copy when resolving per-row p-values from Tukey tables.

## Cycle Literature Screening Workflow

Cycle literature screening is handled independently from the local manuscript
draft. The screening workflow must not read or depend on any local manuscript
files when prioritizing candidates, phrasing evidence, or assigning novelty.

Repository-local builder:

```text
paper/pd/submission/build_cycle_literature.py
```

Primary inputs:

```text
{project}/summary/table/cycle/interval/cycle_timepoint_joint_interval_candidates.csv
{project}/summary/table/cycle/interval/cycle_timepoint_lmer_interval_candidates.csv
{project}/summary/table/cycle/interval/cycle_timepoint_rlmer_interval_candidates.csv
```

Outputs are written under:

```text
{project}/submission/figure&table
```

Generated assets:

```text
cycle_literature_screening.xlsx
cycle_literature_summary.md
cycle_candidate_phase_map.xlsx
```

The screening workbook keeps one row per candidate and records:

- candidate support level: `joint`, `rlmer`, or `lmer`
- metric, band, region, interval bounds, and phase bucket
- evidence type: `direct`, `partial`, `analog`, or `none`
- key literature links and a concise consensus statement
- claim strength: `strong support`, `discussion-only`, or `potentially novel`

Phase buckets follow the current gait-cycle convention:

- `0-18%`: contralateral heel-strike centered early stance
- `18-50%`: ipsilateral swing
- `50-68%`: post-ipsilateral heel-strike transition
- `68-100%`: contralateral swing to the next contralateral heel strike
- wrap intervals such as `97-4%` or `99-11%` are labeled as
  `cycle boundary / contralateral heel-strike centered`

Candidate clustering is driven by the current cycle outputs rather than by
manuscript sections. Primary clusters are:

- `SNr local periodic`
- `SNr local aperiodic`
- `SNr-STN synchrony`
- `SNr->STN directional` when retained `psi` or `trgc` candidates exist

Additional candidate types present in the CSVs, such as `STN` local intervals,
are kept in the workbook as secondary extensions instead of being discarded.
- Figure 1 is allowed to use non-summary assets and placeholder layouts
- boxplot-style panels should prefer `lmer_model` PDFs for display
- main-text quantitative anchors should be retained only when both `lmer` and
  `rlmer` support the result

The manuscript build is toolbox-centered rather than biomarker-centered. The
generated outline organizes results by paradigm type:

- rest medication contrast
- locomotor state contrast
- discrete turn-event alignment
- continuous gait-cycle interval analysis

Electrophysiological narrative order is fixed:

- first: `beta-low`
- second: `beta-high`
- other bands only as supplementary context, extensions, or event-driven
  complements

## Interactive Execution

Execution modules under `paper/pd/merge/` are designed for IPython or editor
cell execution with `# %%` sections instead of CLI entrypoints.

These interactive modules are expected to bootstrap the repository root and
`src/` path automatically so they can be executed directly from an editor
without requiring manual `sys.path` setup first.

Path bootstrap is anchored to the script file itself, for example:

```python
_FILE = Path(__file__).resolve()
_REPO_ROOT = _FILE.parents[3]
```

In this workspace, script-relative discovery is the fixed convention. The
current working directory is not treated as the source of truth for import
setup.
